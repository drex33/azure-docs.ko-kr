# <a name="scale-up-your-azure-devtest-labs-infrastructure"></a>Azure DevTest Labs 인프라 강화
엔터프라이즈 규모에서 DevTest Labs를 구현 하기 전에 몇 가지 주요 의사 결정 사항이 있습니다. 높은 수준에서 이러한 결정 사항을 파악하는 조직은 향후에도 적절한 디자인 관련 결정을 내릴 수 있습니다. 그러나 이러한 사항은 조직에서 개념 증명을 시작 하는 것을 포함 하지 않아야 합니다. 초기 강화 계획에 포함되는 세 가지 주요 영역은 다음과 같습니다.

- 네트워킹 및 보안
- 구독 토폴로지
- 역할 및 책임

## <a name="networking-and-security"></a>네트워킹 및 보안
네트워킹 및 보안은 어떤 조직에서나 운영의 토대가 되는 요소입니다. 엔터프라이즈급 배포를 진행하려면 훨씬 더 심층적인 분석이 필요하지만, 개념 증명을 올바르게 완료하는 데 필요한 요구 사항의 수는 배포보다는 적습니다. 개념 증명 시에 중점적으로 파악해야 하는 몇 가지 주요 영역은 다음과 같습니다.

- **Azure 구독** -DevTest Labs를 배포 하려면 리소스를 만들 수 있는 적절 한 권한이 있는 Azure 구독에 대 한 액세스 권한이 있어야 합니다. 기업계약를 비롯 하 여 Azure 구독에 대 한 액세스 권한을 얻는 방법에는 여러 가지가 있습니다. Azure 구독에 대 한 액세스 권한을 얻는 방법에 대 한 자세한 내용은 [엔터프라이즈에 대 한 Azure 라이선스](https://azure.microsoft.com/pricing/enterprise-agreement/)를 참조 하세요.
- **온-프레미스 리소스 액세스** – DevTest Labs의 리소스가 온-프레미스 리소스에 액세스할 수 있어야 하는 조직도 있습니다. 온-프레미스 환경에서 Azure로의 보안 연결이 필요 합니다. 시작 하기 전에 VPN 또는 Azure Express 경로 연결을 설정 하 고 구성 하는 것이 중요 합니다. 자세한 내용은 [가상 네트워크 개요](../virtual-network/virtual-networks-overview.md)를 참조하세요.
- **기타 보안 요구 사항** – 컴퓨터 정책, 공용 IP 주소에 대 한 액세스 등의 기타 보안 요구 사항은 개념 증명을 구현 하기 전에 검토 해야 할 수 있는 시나리오입니다. 

## <a name="subscription-topology"></a>구독 토폴로지
구독 토폴로지는 Enterprise에 DevTest Labs를 배포할 때 중요 한 디자인 고려 사항입니다. 그러나 개념 증명이 완료 될 때까지 모든 결정을 구체화 해야 하는 것은 아닙니다. 엔터프라이즈 구현에 필요한 구독 수를 평가할 때 고려할 수 있는 두 가지 상반된 방식은 다음과 같습니다. 

- 전체 조직에 구독 하나 사용
- 사용자별 구독 사용

다음으로 각 방법의 이점을 강조 합니다.

### <a name="one-subscription"></a>구독 하나 사용
단일 구독에 대 한 접근 방식은 대기업에서 관리할 수도 없습니다. 그러나 구독 수를 제한하는 경우에는 다음과 같은 이점이 제공됩니다.

- **예측** - 기업의 구독 비용을 예측할 수 있습니다.  구독이 하나이면 모든 리소스가 단일 풀에 포함되므로 예산을 책정하기가 훨씬 쉬워집니다. 이 방식에서는 청구 주기의 특정 시점에 비용 제어 조치를 취할 시기를 더 간단하게 결정할 수 있습니다.
- 모든 업데이트는 여러 구독에서 업데이트 하는 것이 아니라 하나의 구독에만 필요 하기 때문에 Vm, 아티팩트, 수식, 네트워크 구성, 권한 및 정책의 **관리 효율성이** 더 쉬워졌습니다.
- **네트워킹** 활동은 온-프레미스 연결을 요구 하는 기업에 대 한 단일 구독에서 더 간단 합니다. 구독 (허브-스포크 모델)을 통해 가상 네트워크를 연결 하려면 추가 된 구독과 함께 필요한 구성, 관리 및 IP 주소 공간이 더 필요 합니다.
- 모든 사람이 동일한 구독에서 작업 하는 경우 **팀 공동 작업** 을 더 쉽게 수행할 수 있습니다. 예를 들어 VM을 동료에 게 다시 할당 하거나 팀 리소스를 공유 하는 것이 더 쉽습니다.

### <a name="subscription-per-user"></a>사용자별 구독 사용
사용자별로 별도의 구독을 사용하는 방식 역시 단일 구독과 동일한 기회를 제공합니다. 여러 구독을 사용하는 경우의 이점은 다음과 같습니다.

- **Azure 크기 조정 할당량** 은 채택을 방해 하지 않습니다. 예를 들어 이 문서 작성 시점에 Azure에서는 구독당 스토리지 계정 200개를 포함할 수 있습니다. 대부분의 Azure 서비스에 대 한 작업 할당량이 있습니다 (대부분 사용자 지정 가능, 일부는 불가능). 사용자 당이 구독 모델에서는 대부분의 할당량에 도달할 가능성이 매우 낮습니다. 현재 Azure 크기 조정 할당량에 대한 자세한 내용은 [Azure 구독 및 서비스 제한, 할당량 및 제약 조건](../azure-resource-manager/management/azure-subscription-service-limits.md)을 참조하세요.
- **지불 거절** - 조직에서 현재 모델을 사용해 비용을 처리할 수 있으므로 그룹이나 개별 개발자에 대한 지불 거절이 훨씬 쉬워집니다.
- **소유권 및 권한** - DevTest Labs의 환경에서는 단순한 권한과 소유권이 사용됩니다. 즉, 개발자에게 구독 수준 액세스 권한을 제공하며 네트워킹 구성, 랩 정책, VM 관리 등의 모든 작업에 대해서는 전적으로 개발자가 책임을 집니다.

하지만 기업에서는 제약 조건이 많아 위의 두 가지 극단적 방식을 사용하기가 어려울 수 있습니다. 그러므로 이 두 방식의 중간에 해당하는 방식으로 구독을 설정해야 할 수 있습니다. 최상의 방법으로 조직의 목표는 가능한 최소 구독 수를 사용 하는 것입니다. 전체 구독 수를 늘리는 강제 함수를 염두에 두어야 합니다. 이를 위해 구독 토폴로지는 DevTest Labs의 엔터프라이즈 배포에 중요 하지만 개념 증명을 연기 해서는 안 됩니다. 조직에서 구독 및 랩 세분성을 결정 하는 방법에 대 한 자세한 내용은 [거 버 넌 스](devtest-lab-guidance-governance-policy-compliance.md) 문서에 나와 있습니다.

## <a name="roles-and-responsibilities"></a>역할 및 책임
DevTest Lab 개념 증명에는 각기 책임이 정의된 세 가지 기본 역할(구독 소유자, DevTest Labs 소유자, DevTest Labs 사용자)이 포함되며 필요에 따라 참가자도 포함될 수 있습니다.

- **구독 소유자** – 구독 소유자는 사용자 할당, 정책 관리, 네트워킹 토폴로지 만들기 및 관리, 할당량 늘리기 요청 등 Azure 구독을 관리할 수 있는 권한을 가집니다. 자세한 내용은 [이 문서](../role-based-access-control/rbac-and-directory-admin-roles.md)를 참조하세요.
- **DevTest Labs 소유자** - DevTest Labs 소유자에게는 랩에 대한 모든 관리 권한이 있습니다. 이 사용자는 사용자 추가/제거, 비용 설정 관리, 일반 랩 설정, 기타 VM/아티팩트 기반 작업을 담당합니다. 랩 소유자에게는 모든 DevTest Labs 사용자 권한도 있습니다.
- **DevTest Labs 사용자** – DevTest Labs 사용자는 랩에서 가상 머신을 만들고 사용할 수 있습니다. 이 개별 사용자에게는 직접 만드는 VM에 대한 최소한의 관리 기능(VM 시작/중지/삭제/구성)이 제공됩니다. 하지만 다른 사용자의 VM을 관리할 수는 없습니다.
